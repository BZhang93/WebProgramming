  5. Conclusion - You can also list future considerations and recommendations with costs.

Frequently Asked Questions

Q: Are we supposed to read the downloaded source code in order to make our attacks on the site we are attacking? Or should we try to do it without going through the source code? I'm not sure how to tell my attacks are working.

A: Two step process:

  1. First, perform attacks WITHOUT reading the source code! This is also known as "black box" testing.
  2. Then, download or get from your partner, the source code of the application. Peruse through it, perform a code review of it (by hand and/or by a scanner such as RATS), and document for your findings.

If you find anything glaring, then go back to the application and attack it. This is also known as "white box" testing. For example, after you read the source code for the search function and you are suspicious that you can perform a SQL injection attack to delete a table (also given the database structure from your partner), go ahead an try it out :-)

Q: Is it necessary to use all the programs listed above? I'm having trouble running Burp suite and RATS specifically. I was able to get a good amount of information from Paros and from looking at the source code, though.

A: No.

Q: I'm using Paros and it gave me back a URL under alerts. Does that mean it tried a SQL injection and worked?

A: Perhaps. It could also be a false positive. You have to check in the database (in this case, ask your partner), if it went through. However, this is something that you can document.

Q: If Paros doesn't give me back any alerts when spidering through something, does that mean that it hasn't found any, or that I don't know how to use the application?

A: If Paros doesn't return anything, it didn't find anything. But any good security practitioner would know that you cannot depend on a (especially just one) scanning tool, run it, and call it a day. I believe that the XSS that I performed on Sarah's application was not detected in Paros --so that should tell you something. Now, a more reliable and ruthless method would be to use IBM (formerly Watchfire's) AppScan: http://www-01.ibm.com/software/awdtools/appscan/. It is not free, but arguably the best scanner money can buy.

Q: I can't find any problems with my partner's application. What should I do?

A: You can make recommendations (and I find it hard to believe that you cannot find problems). Example: if you notice that your partner's web application uses the insecure methods, you can recommend to upgrade application to use more secure options.
